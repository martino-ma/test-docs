# 电商系统测试策略

## 1. 优先级与取舍

- **P0（最高优先级）**：支付、库存、订单状态。任何异常都会直接导致资损或用户投诉。
- **P1**：发货、退货、库存回滚。虽影响用户体验，但可异步修复。
- **P2**：部分发货、秒杀场景。虽然重要，但可通过限流和监控兜底。

**取舍决策**：  
在时间有限的情况下，优先保障强一致性场景（支付 + 扣库存）的自动化回归和补偿覆盖。对于最终一致性场景（如 WMS 同步），通过消息重试和对账日志来保证，不要求立即完美。

## 2. 库存同步一致性测试策略

**核心原则**：所有库存变更操作必须幂等，且支持重试。

**测试方法**：

- a. 模拟 MQ 重复发送同一消息，验证库存只变化一次。
- b. 模拟消息乱序（先到扣减再到占用），验证系统能否通过状态机过滤或版本号机制拒绝非法请求。
- c. 使用 JMeter 并发调用取消订单 API，验证库存恢复的原子性。

**对账**：每日运行 SQL 脚本，比对订单表与库存日志表，输出差异报告。

## 3. 异常处理与补偿策略

| 异常类型 | 处理方式 | 补偿策略 |
| :--- | :--- | :--- |
| 支付回调超时 | 异步重试 + 主动查询 | 若最终支付成功但订单未推进，由补偿 Job 修复 |
| WMS 占用失败 | 直接返回失败，订单创建失败 | 无需补偿，用户重试 |
| 扣减库存超时 | 进入死信队列 + 人工介入 | 巡检工单，财务对账 |
| 取消订单时库存恢复失败 | 记录失败日志 + 定时重试 | 每小时扫描未恢复的订单 |

**补偿 Job 设计**：  
每分钟扫描“支付成功超过 5 分钟未扣库存”的订单，重新调用 WMS 扣减接口（幂等性）。

## 4. 发布准入/准出解释

- **准入**：强调功能与性能的基线，避免未达标版本进入测试。
- **准出**：强调生产可观测性与应急能力，确保上线后问题可发现、可恢复。
- **回滚预案**：包含数据库版本回滚脚本、应用镜像切换、开关降级等。
